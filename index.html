1 · The pipeline

Manufacturing datasets rarely come with segmentation masks. The core idea here: train only a box detector (cheap labels), then let a promptable segmenter turn each box into a mask at inference time. Each YOLO box is fed to SAM as a box prompt — top-left and bottom-right corners become a 2-point prompt — and SAM returns the defect mask, zero-shot.

image→ YOLOv8s (boxes + classes)→ box corners as prompts→ SAM / MobileSAM (masks)→ per-class overlay

Two implementations of the same design:

2 · Dataset — and the hunt for the labels

The images come from a public bearing-defect dataset by Kangning Li (figshare, CC0): 5,824 processed 554×416 images across three surface-defect classes — aocao (groove), cashang (scratch), huahen (scratch-mark). The catch: the dataset article contains no annotations at all.

The labels were hiding in the code. All three figshare articles share one author. The source paper ("Bearing defect detection based on the improved YOLOv5 algorithm," PLOS One 2024) says labels were made with LabelImg — and archive inspection revealed a zhou/ folder inside the author's YOLOv3/YOLOv5 code archives holding the complete labeled split: 4,660 train / 582 val image-label pairs in YOLO txt format, three balanced classes. No hand-annotation was needed after all.

3 · Training the detector (GB10)

YOLOv8s fine-tuned from COCO weights — 640 px, batch 32, AdamW, 100 epochs (1.58 h on the GB10). One hardware lesson first: the GB10 is compute capability sm_121, and PyTorch's CUDA-12.8 wheels crash in nvrtc JIT on it — the CUDA-13 (cu130) build is mandatory on this machine.

4 · The GB10 benchmark: YOLO + SAM ViT-H

On the workstation, each YOLO box is handed to SAM 1.0 (ViT-H checkpoint, 2.4 GB) as a batched box prompt — one image-encoder pass per image, all boxes decoded in a single call. This full-precision pipeline is the project's quality reference for everything that follows.

5 · Porting to the IQ-9075 NPU

The target: run everything heavy on the board's Hexagon HTP (hexagon v73, soc id 77). Three realities shaped the port:

ViT-H cannot run on an NPU

A 637 M-parameter encoder is far beyond HTP budgets. Swapped for MobileSAM (5 M TinyViT encoder, near-equivalent masks). Box prompts survive the swap: corners → 2 labeled points.

HTP compilation needs Qualcomm silicon

Local yolo export format=qnn dies on non-Qualcomm hosts. Everything was compiled in the Qualcomm AI Hub cloud for the exact device — which also profiles on a real hosted IQ-9075.

Runtime = ONNX Runtime QNN EP

AI Hub's precompiled_qnn_onnx wraps each HTP context binary in an ONNX file — loadable from Python via onnxruntime-qnn. On ORT ≥ 1.24 the QNN EP is a plugin that must be explicitly registered.

best.pt→ ONNX (sanitized)→ AI Hub compile → HTP context binary→ ONNX Runtime QNN EP on the board

6 · The quantization lesson the key finding

The first production build quantized YOLO to W8A16 (INT8 weights / INT16 activations, 100 calibration images) — the standard recipe for using an NPU's full INT8 throughput. It compiled cleanly, ran fully on the HTP… and detected almost nothing.

Failure signature: every class confidence collapsed to exactly 0.50 — sigmoid(0) — versus 0.90–0.93 in full precision. Reproduced with the quantized QDQ model on a plain CPU: the model itself was broken before it ever reached the NPU.

Root cause. The Ultralytics YOLOv8 export is an end-to-end graph whose head concatenates box coordinates (range 0–640) and class probabilities (range 0–1) into one output tensor — post-training quantization must pick a single scale (0.0104 for range [0, 682]) for both, and the DFL/anchor decode arithmetic gets quantized with it. The class-score information is crushed. This is a known YOLO PTQ failure mode; Qualcomm's own yolov8_det recipe restructures the head before quantizing for exactly this reason.

Fix. Ship the detector in FP16 instead (the HTP supports FP16 natively; FP32 never). Verified end-to-end on a hosted IQ-9075 with real bearing images before redeploying:

Two more defects were caught in the same investigation: the EPContext model initially exposed uint16 IO (compiled with --quantize_io; scale/zero-point unrecoverable on the board — recompiled with float32 IO), and the runner's NMS passed xyxy boxes to an API expecting xywh. Both fixed and committed.

Reading the numbers with the images: detection agrees almost perfectly — boxes land in the same places with confidences within ~0.01, and per-class defect totals agree within 5–10%. The residual IoU gap is dominated by mask-shape differences between SAM ViT-H and MobileSAM (expected by design: 637 M vs 5 M encoder) and by IoU's sensitivity on tiny masks — a few boundary pixels on a small groove crush the score even when both models mark the same defect.

With YOLO at ~10–25 ms, the full pipeline lands around 140–160 ms per image (~6–7 fps), entirely on the NPU — encoder once per image plus 6 ms per detected defect. PSNR ≥ 30 dB is the usual "good" bar; 47–60 dB means the HTP output is numerically near-identical to desktop FP32.

9 · Lessons learned

PTQ can silently destroy a YOLO head

Mixed-range concat outputs (boxes 0–640 + probs 0–1) under one quant scale collapse class scores. Test the quantized model off-device first — a CPU run of the QDQ graph reproduces it in seconds.

FP16 on modern HTP is a legitimate production path

Zero calibration, near-zero accuracy loss, still 100% on-NPU. INT8 is an optimization, not a prerequisite.

Compile in the cloud, not on the host

QNN's HTP backend won't initialize on non-Qualcomm machines. AI Hub compiles for the exact SoC and profiles on real hardware before you touch the board.

Check tensor layouts on compiled artifacts

AI-Hub MobileSAM EPContext models are NHWC (channels-last); the YOLO one stays NCHW. Assume nothing — read the ONNX IO.

Don't ship --quantize_io to a float pipeline

Quantized IO exposes uint16 tensors whose scale/zero-point the board code can't recover. Keep model IO float32; let the context convert internally.

Labels may exist where you least expect

An "images-only" dataset had its full YOLO label set hiding inside the author's code archives. Read the paper, inspect the archives, then annotate.
